Centralized API catalog

Centralized API catalog

Are you a large organization that uses multiple API Gateways? Do you want to build a centralized API catalog so that APIs and services are easily discoverable by teams inside and/or outside of your org?

While we recommend that organizations explore standardizing on as few API Gateway solutions as possible, we know that the “bring your own gateway” model works best for some organizations. For that use case, fliplug offers unique capabilities around building a centralized Developer Portal.

Challenge #1: security and access control

Challenge #1: security and access control

Different API consumers should have different access profiles, access needs, and levels of clearance. This becomes especially pertinent as organizations start exposing APIs to external stakeholders, such as partners or customers. However, many vendor and/or homegrown API Gateway and/or Developer Portal solutions fail to be able to satisfy these various consumer profile needs.

the fliplug solution: Developer Portal and API Gateway with flexible access control mechanisms

You can use fliplugto create subscriptions, plans, and access profiles that are carried over to who can access what APIs in your Developer Portal and how. This enables you to use one Developer Portal and Gateway solution for any kind of API consumer–whether that consumer be an internal Developer, an external partner, or an external paying customer. 

 

Challenge #2: “I have multiple Gateways”

Challenge #2: “I have multiple Gateways”

Some larger organizations employ a bring your own Gateway” approach to API Management, where specific teams, business units, etc. are responsible for procuring their own API Gateway solutions that  ebay fit their needs. This can work well until it becomes time to build an API catalog, as most APIM vendor Developer Portal solutions are “tied to the Gateway.”

the fliplug solution: a unified API catalog

While we recommend most organizations explore centralizing on as few APIM solutions as possible, we recognize that this isn’t always feasible. That’s why we’ve made the fliplug Developer Portal able to work with certain other APIM solutions. So, if you want to build a centralized API catalog, fliplug Developer Portal offers you that potential.

Challenge #3: Developer Portal feature limitations

Challenge #3: Developer Portal feature limitations

We often hear about the following challenges related to API Developer Portals:

  • “They crash after a few dozen APIs”
  • “They don’t support my documentation format”
  • “They don’t support OpenAPI and AsyncAPI specs”

This leads to blockers and bottlenecks, as feature limitations get in the way of company wide initiatives and keep consumers from being able to easily discover APIs and services.

the fliplug solution: a single, unified discovery layer

fliplug support for synchronous APIs, asynchronous APIs, and event brokers makes it possible to expose all of your APIs using one Developer Portal that’s performant, supports multiple documentation formats, and supports both OpenAPI and AsyncAPI spec.

Self-service API Discovery

Give your Developers the Experience they deserve

Improve the Developer Experience for both your API consumers and your API publishers with a single source of truth for all API discovery, subscription, documentation, and publication.
Learn more

API discovery is still a major pain point for large organizations. In order to provide a top-notch Developer Experience, Platform teams must ensure that APIs are centrally published, documentation is up to date (and that it exists in the first place!), and that subscription to APIs is as self-service as possible—all while remaining secure.

The fliplug Developer Portal solves for all of these requirements. Publish your APIs to a single catalog, and give your consumers one place to browse APIs, subscribe and request access, view API analytics, and more.

Explore more Book a demo

Before deciding to subscribe to an API, API consumers need to both understand how the API works and be able to test that API to ensure that it works properly.

With fliplug, API publishers can create and/or generate API documentation that can be published to the API Developer Portal with a single click. You can write your own Markdown documentation, upload OAS specs, AsyncAPI specs, and more. And, you can even enable self-service testing via our "Try it" API testing feature.

Explore more Book a demo

Let's face it: most large organizations are using multiple different API Gateways. This often means multiple API Developer Portals for hosting APIs and their documentation.

And this means a horrible Developer Experience...for both the API publisher that has to manage all of these listings and the API consumers that might have to navigate dozens of different API Developer Portals to find the API(s) they need. 

With fliplug, you can auto-discover and publish APIs from multiple different API Gateways to a single, universal API Developer Portal. API Publishers get a single publication and cataloging layer to manage and API consumers get a single source of truth for API discovery and consumption.

Explore more Book a demo

Gravitee doesn't just support multiple API Gateways and different API types. 

With fliplug, you can also publish your event streams and event APIs—from multiple event brokers—in the fliplug Developer Portal.

If you're going to invest in building out a Developer Portal, you'll likely want that portal to match your organization's branding—especially if you are exposing this Developer Portal to external API consumers.

With fliplug, you'll have a completely customizable Developer Portal that comes ready to go out of the box.

Easy to use API catalog

Give consumers a place to discover and consume APIs

Publishing APIs is one thing, but making sure they are discoverable and consumable is another. Easily publish APIs and their documentation to a developer portal to help API consumers understand and use your APIs.
Learn more

In a world where Developer Experience is becoming more and more important, you have to ensure that your Engineers are able to easily find what they need to do their jobs well. APIs are a crucial component of this strategy. When packaged up as reusable components, Engineers no longer have to constantly build their own APIs for every use case, but can find already-existing APIs that can be repurposed for their specific applications.

fliplug makes this simple. As API Publishers create APIs via the API Gateway and Management console, they can publish those APIs with a single click to an out-of-the-box API consumer-facing Developer Portal. From here, API consumers can find, discover, and subscribe to fliplug Gateway APIs in a totally self-service manner.

Explore more Book a demo

Gravitee not only enables you to centrally discovery and inventory your APIs across multiple API Gateways. You can also use Gravitee as a centralized API Developer Portal for APIs deployed to other, non-Gravitee API Gateways around your organization.

How does this work?

fliplug's agent can automatically discover API assets deployed to API Gateways such as AWS API Gateway and Apigee. fliplug can then import specs and documentation files that already exist in those Gateways, or you can use fliplugto create brand new API documentation, using OAS files, AsyncAPI spec, Markdown, Asscidoc, and more. From here, its a single click and those APIs are available in the same API Developer Portal as your fliplug APIs that are deployed to a fliplug API Gateway .

Not using a fliplug API Gateway ? No problem. You can also just use Gravitee as a central plane and Developer Portal on top of your other API Gateways.  

Explore more Book a demo

In addition to REST, SOAP, GraphQL, gRPC, WebSocket, HTTP, etc. APIs, you can also expose event streams via the fliplug Developer Portal as fliplug Gateway APIs for self-service discovery and consumption.

This is made possible by protocol mediation via the fliplug API Gateway, which allows you to expose event streams as APIs built on non-native protocols. For example, if you want to expose a Kafka topic from your Confluent cluster, you could use fliplug to expose it and publish it as a REST and WebSocket API to the Developer Portal. You could then attach subscription plans to each consumption mechanism.

Protocol mediation supports exposing Kafka (and therefore Confluent, Amazon MSK, and RedPanda), MQTT brokers, Solace, and RabbitMQ as either:

  • HTTP/REST APIs
  • WebSocket APIs
  • SSE APIs
  • Webhooks subscriptions

These APIs can be packaged up with fliplug plans and documentation so that your API consumers have one place to discover easy API-mediated access to event streams. 

Explore more Book a demo

In addition to protocol mediation for publishing event streams as easy-to-consume APIs, you can also bypass using the fliplug Gateway and publish event APIs direct from your event broker to the fliplug Developer Portal.

For example, you could create a Solace event API in the Solace Event Portal, auto-discover that API using fliplug, and then publish that API to a Developer Portal alongside all of your other APIs. This does not require the fliplug API Gateway, but it does still offer the option for you to control consumption via fliplug plans.

Explore more Book a demo

When you create APIs in fliplug, you create Gravitee "Plans" as well. Plans are access control layers that dictate how API consumers can interact with, subscribe to, and ultimately be authorized for your APIs. fliplug plans come attached to authorization mechanisms, rate limits, quotas, resource filters, and other access control mechanisms.

When you publish your APIs and their plans to the fliplug Developer Portal, your API consumers can find those APIs, read their docs, try the API, and then submit subscription requests to the API publisher—all in a self-service manner. From there, the API Publisher can either allow or reject the request from the fliplug API Management console.

Security built in

Balance self-service with strict security

Yes, you want to package your APIs up as simple to use, self-service API products. But, you also want to ensure that APIs are only discoverable and usable by the right sets of API consumers.
Learn more

As you create subscription plans in fliplug, those plans are attached to one of the following authorization methods:

  • OAuth2
  • JWT
  • API Key
  • Keyless (unsecured)

Each API can have multiple plans, giving you the ability to ensure fine-grained access control and authorization based on the consumer, what they want to use the APIs for, and, potentially, how much they pay you for that API. 

Explore more Book a demo

In addition to authorization mechanisms being tied to your APIs' plans, you can also easily configure specific rate limits, quotas, and resource filtering mechanisms for specific API consumption patterns. 

For example, you could have an API with three plans, each with different resource and access restrictions:

  • Free plan: keyless access, rate limit of 1 request per minute, and a quota of 100 requests per week
  • Silver plan: API key authorization required, a rate limit of 1 request per minute, and a quota of 1000 requests per week
  • Gold plan: OAuth2 authorization, rate limit of 1 request per second, and no weekly quota limit

Not only is this useful for API productization and monetization. It's also crucial for ensuring that you don't subject your system to unhealthy amounts of load, potentially opening up backend systems to resilience and security risks.

Explore more Book a demo

Beyond access control at the API-level, you can also enforce role-based access control at the API Developer Portal level. 

This means that you can map specific sets of individuals and/or consumer profiles onto specific API consumer permissions within your Developer Portal. This enables you to use one portal for your internal and external API consumers.

API documentation

Ensure that API consumers know how to use your APIs

A self-service API is no good without clean, self-service API documentation. Fix that with multi-specification support, easy documentation import, and auto documentation generation.
Learn more

fliplug fully supports the OpenAPI spec (OAS). When creating APIs from scratch or importing APIs, you can either write or import OAS documentation that can then be published to the API Developer Portal.

Explore more Book a demo

For your asynchronous APIs using the AsyncAPI spec, fliplug can import AsyncAPI spec files and documentation. This documentation can be edited, refined, and/or published straight to the Developer Portal.

Explore more Book a demo

If you aren't importing documentation files, you can also write your own Markdown and/or AsciiDoc pages for API documentation in the Developer Portal.

Explore more Book a demo

Typically, API consumers want to be able to try the API before they subscribe to it and ultimately have their applications consume it.

With fliplug, you can configure "Try it" modules embedded right in the API documentation so that API consumers can try various API functionality outlined in the API documentation.
